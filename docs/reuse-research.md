# The Delve — Open-Source Reuse Research Brief

A vetted map of Godot 4 repositories to borrow from while building The Delve's next subsystems (CombatSim, offline progress, gacha, talent tree, equipment paperdoll, inventory, iso combat). Every repo below was verified by inspecting the actual source/API, not just the README. Grouped by CLAUDE.md build order, with concrete adapt-to-our-stack notes (GameState / EventBus / SaveManager autoloads, typed GDScript everywhere).

> **The hard truth up front:** none of these gives you a drop-in deterministic, headless, tick-based party CombatSim — the project's single most important architectural rule (§3). That core must be built fresh. What these repos *do* give you: the surrounding scaffolding (saves, big-number utils, loot tables, stat aggregation, talent graphs, inventory containers, UI/theme patterns, a test harness) and high-quality structural references.

## Top picks

1. **[gdquest-demos/godot-open-rpg](https://github.com/gdquest-demos/godot-open-rpg)** (MIT, ~2.8k★, Godot 4.6, active) — The highest-quality, most-maintained Godot 4 typed-GDScript RPG reference. Best for combat-modeling concepts, stat aggregation, inventory, and modern project structure. Clone branch `main`.
2. **[ninetailsrabbit/Lootie](https://github.com/ninetailsrabbit/Lootie)** (MIT, ~114★, Godot 4.3/4.4) — Mature, typed, data-driven loot library with weight + rarity-tier + percentage modes and **seed-based determinism** — the latter is gold for headless offline-progress reproducibility. The closest single match for gacha/loot rolls (layer your own pity on top).
3. **[peter-kish/gloot](https://github.com/peter-kish/gloot)** (MIT, 927★, Godot 4.4+) — Production-grade inventory: `Inventory`/`InventoryItem`/`ItemSlot`, JSON item prototypes with inheritance, stacking, serialization. Backs both the equipment paperdoll (one `ItemSlot` per slot) and the loot/consumable bag directly.
4. **[don-tnowe/godot-worldmap-builder](https://github.com/don-tnowe/godot-worldmap-builder)** (MIT, 135★, Godot 4) — The strongest talent-tree fit: allocatable, dependency-gated node graph with in-editor authoring, already-typed allocation/prereq/refund/save API, and glow-state styling. Wraps cleanly into our StatBlock + SaveManager.
5. **[TinyTakinTeller/GodotProjectZero](https://github.com/TinyTakinTeller/GodotProjectZero)** (MIT code, ~183★, Godot 4.4) — The only shipped, actively maintained, MIT idle game on our exact engine. Best reference for offline-timing/clamp/welcome-back scaffolding, multi-slot JSON saves, autoload organization, and big-number utils. (Assets are CC-BY-NC-SA — code only.)

Honorable mentions: **[bitwes/Gut](https://github.com/bitwes/Gut)** is the de-facto test framework and a perfect fit for asserting on a deterministic sim; **[youssof20/savestate](https://github.com/youssof20/savestate)** has directly liftable atomic-write + schema-migration save internals.

## Save / State

The brief mandates JSON in `user://` + a last-played UTC timestamp + (implicitly) versioning and crash-safety. We already have GameState/SaveManager/EventBus, so these are mostly internals-to-lift and pattern references.

- **[youssof20/savestate (SaveState Lite)](https://github.com/youssof20/savestate)** — **MIT** — **Best Save lift.** Copy the **atomic write pattern (write temp → validate → rename) + rolling `.bak` backup** and the **schema-version + forward-merge migration** into our typed `SaveManager`. It uses stable Godot 4 time APIs but has **no built-in "last-played" field** — add `last_played_utc` yourself. Wrap its Variant key/value store behind typed accessors (`save_profile(state) -> void`, `load_profile() -> GameStateData`). Small project (~13★), so lift-and-own the patterns rather than depend on the addon.
- **[EiTaNBaRiBoA/ForgeJSONGD](https://github.com/EiTaNBaRiBoA/ForgeJSONGD)** — **MIT** — Serialization layer *beneath* SaveManager. Auto-serializes `class_name` objects to/from JSON, handling Vector2 (talent positions) and Color (rarity borders) natively, killing `to_dict`/`from_dict` boilerplate for StatBlock/Hero/Item/etc. **Constraint:** deserialization is type-directed (caller passes the target type) — for polymorphic collections (mixed Item subclasses), store an explicit `"type"` field and dispatch yourself. Set `only_exported_values = true` so transient sim state never leaks into saves. Note its `str_to_var` injection caveat for untrusted/cloud saves.
- **[fernforestgames/godot-savekit](https://github.com/fernforestgames/godot-savekit)** — **MIT** — Targets exactly Godot 4.5+. Borrow the **pluggable serializer interface** (`SaveKitSerializer`/`SaveKitDeserializer`) so JSON-in-dev / binary-in-release is a swap, not a rewrite — lift the interface into our existing SaveManager rather than adding a second SaveManager autoload. Very young (~12★, v0.1); reference, not dependency.
- **[AdamKormos/SaveMadeEasy](https://github.com/AdamKormos/SaveMadeEasy)** — **MIT** — Most-popular Godot 4 save addon (~218★). Use as an API-ergonomics + **autoload-boot-first** reference. Its **OS-ID-keyed encryption** idea is worth adopting for Steam tamper-resistance — but gate it behind a build flag (it breaks cloud-save portability across machines). Its untyped Variant bag conflicts with our typed-resource discipline; don't adopt wholesale.
- **[godotengine/godot-docs — saving_games.rst](https://github.com/godotengine/godot-docs)** — **CC-BY-3.0 (prose; `classes/` is MIT)** — The authoritative baseline for `FileAccess` + `JSON.stringify`/`parse` + the `to_dict()`/`from_dict()` contract. **Drop** its per-line, scene-tree "Persist" node-group approach — our state lives in GameState data, not persisted nodes. Layer on what the tutorial omits: `save_version` + migration, `last_played_utc`, and atomic write.
- **[derkork/godot-safe-resource-loader](https://github.com/derkork/godot-safe-resource-loader)** — **MIT** — Security hardening **only**, and **LOW relevance** for us: plain JSON parsed with `JSON.parse_string()` can't execute embedded scripts, so the `.tres`-injection threat this defends against doesn't exist on our JSON-first path. Bookmark *only* if you ever load `.tres`/shared saves from `user://`.

## CombatSim

No repo provides a deterministic headless tick sim — all combat references here are real-time/turn-based and rendering-coupled. Use them for combatant/stat **modeling** and re-architect the loop yourself.

- **[guladam/godot_autobattler_course](https://github.com/guladam/godot_autobattler_course)** — **MIT** — Closest genre match (autobattler, Godot 4.5, typed, ~178★, active). **Most valuable piece: the `modifier.gd`/`modifier_handler.gd`/`modifier_value.gd` trio** — a clean additive/multiplicative stat-modifier stacking system adaptable almost verbatim into `StatBlock.gd` to layer hero base + gear + pets + relics + talents + Team Aura. The FSM (`finite_state_machine.gd`/`state.gd`) is reusable for hero AI states (seek/attack/heal/retreat) — but as a **logical** state machine in the sim, not the Node2D version here. **Do NOT** lift its combat loop: it's real-time, Node2D-, projectile-, and collision-driven — it cannot run headless/fast, which contradicts §3.
- **[gdquest-demos/godot-open-rpg](https://github.com/gdquest-demos/godot-open-rpg)** — **MIT** — Reuse the **conceptual combat model** (Combatant entities, stat-driven damage/healing resolution, action ordering) and its stat → derived-stat aggregation, but its combat is turn-based and `await`/animation-coupled. Re-architect: replace the turn queue with a fixed logical tick loop, strip all animation coupling so N ticks run headless. Clone branch `main`.
- **[guladam/godot_autobattler_reference](https://github.com/guladam/godot_autobattler_reference)** — **MIT** — Same author, 3D presentation. Value is purely the **sim-vs-presentation separation pattern** (state in `components/`, scenes merely visualize) and data-driven units/abilities — reinforces our §3 rule. Don't copy its 3D scenes.
- **[Cute-Fame-Studio/3D-TurnBasedCombat](https://github.com/TheRealFame/Turn-Based-Combat)** — **MIT** — **LOW relevance** (downgraded from the candidate's "medium"). Turn-based, animation-driven, RPG-Maker-style — the opposite paradigm to ours. The only transferable idea is the ~20-line base-battler inheritance pattern. A 10-minute architecture skim at most; not worth integrating.
- **[Zennyth/EnhancedStat](https://github.com/Zennyth/EnhancedStat)** — **MIT** — **StatBlock design reference** (Godot 4.1, dormant — do *not* vendor; it pulls an `enhanced_common` dep). Models exactly §7's pattern: base + Modifiers {FLAT_ADD, PERCENT_MULT, CUSTOM} + reactive recompute, with "Aggregated stats" mapping onto derived HP/ATK/DEF/crit. Reimplement typed; resolve order = sum flats then apply mults; route stat-changed signals through **our EventBus**; ensure recompute is synchronous/order-stable so offline ticks reproduce exactly.
- **[ChronoDK/GodotBigNumberClass](https://github.com/ChronoDK/GodotBigNumberClass)** — **MIT** — The de-facto idle big-number lib (~195★). Solves deep-stage HP/gold scaling beyond float64 and idle formatting (`toAA` → "4.56AA", `toMetricName` → "2.00M"). It's **untyped**, so vendor `Big.gd` into `systems/data/` as `BigNumber` (`class_name BigNumber extends RefCounted`), add type hints to public signatures, keep the proven mantissa/exponent math intact. It's `RefCounted` (per-call allocations) — prefer the in-place `*Equals` variants in hot loops. **First confirm you actually need >1e15 magnitudes** — if the stage cap keeps numbers small, a plain int64/float + a tiny formatter avoids the dependency.

## Idle / offline progress

- **[TinyTakinTeller/GodotProjectZero](https://github.com/TinyTakinTeller/GodotProjectZero)** — **MIT (code); assets CC-BY-NC-SA — DO NOT ship assets** — **Highest-value idle reference.** `global/utils/date_time_utils.gd` `unix_delta(start, end)` is the exact offline timestamp-delta primitive; `scenes/controller/offline_controller/offline_controller.gd` computes output for a "cycles" count **headlessly**; `offline_screen.gd` is the welcome-back UI. **Adapt:** keep the elapsed-delta → ticks math and welcome-back scaffolding; replace their resource-production loop with `CombatSim.advance(n_ticks)`; add the 12h clamp (their `global/const/limits.gd` shows where). Reuse the timing/clamp/welcome-back layer, not the production formulas (genre mismatch — it's a clicker, not a party ARPG). Also see Save/State and UI sections for its multi-slot JSON save and autoload patterns.
- **[ChronoDK/GodotBigNumberClass](https://github.com/ChronoDK/GodotBigNumberClass)** — **MIT** — Big-number formatting/arithmetic for offline gold accrual and the auto-loot ticker. Serialize each value as its `{mantissa, exponent}` pair in saves rather than a lossy float. (Full notes under CombatSim.)
- **[veprogames/godot-bignumber](https://github.com/veprogames/godot-bignumber)** — **MIT** — **LOW relevance / archived (Feb 2026).** A clean, typed `break_infinity.js`-style mantissa+exponent reference. The Delve as specified (modest numbers, +18% auras, 12h cap, no prestige) doesn't approach 1e308. Keep it as an instructive reference *only if* a deep prestige/ascension layer is added later.

## Gacha / loot

The weighted-draw engine is well-covered; **the pity counter is on you** in every case — Gacha (§7) needs pulls-since-rarity tracking and a forced high-rarity pull at the pity threshold.

- **[ninetailsrabbit/Lootie](https://github.com/ninetailsrabbit/Lootie)** — **MIT** — **Best loot/gacha fit** (~114★, typed, Godot 4.3/4.4, last release Oct 2024). `LootieTable` + `LootItem` + three probability resources (`LootItemWeight`/`LootItemRarity`/`LootItemChance`) and a 7-mode enum (Weight, RollTier, PercentageProbability, + combined). Authors loot as `.tres` in `res://data/` (matches our data-driven rule). **`seed_value`/`seed_string` determinism** is the standout: the headless sim and offline catch-up can produce identical loot, avoiding desync — seed each batch from last-played timestamp + run counter. `should_drop_always` = guaranteed daily-dungeon rewards; `is_unique` = one-time unlocks. **No pity counter** — layer it on by injecting a guaranteed high-rarity item once pity hits. Low adapt effort (already typed Godot 4).
- **[PieKing1215/godot-loot-tables](https://github.com/PieKing1215/godot-loot-tables)** — **MIT** — Clean three-layer Resource pattern (`PoolEntry` → `Pool` → `LootTable`) — exactly the `.tres` nesting our `res://data/` wants for stage loot (gold pool + item pool + rare pool). Dormant/WIP (2★), so reimplement as a **typed pattern** with `class_name` and `roll() -> Array[Resource]`. **Critical:** it likely uses global `randi()` — you **must** thread a seeded `RandomNumberGenerator` through `roll()` for §3 determinism. Skip its WIP graph editor.
- **[robertcorponoi/godot-loot-system](https://github.com/robertcorponoi/godot-loot-system)** — **MIT** — Compact (~11★, 2 scripts) reference for the weight-tally selection (the gacha core), nested tables as rarity tiers, `should_drop_always`, and `is_unique`. **No pity, no UI, no rarity colors.** Rewrite into typed Resources with `class_name`/`@export`/`Array[T]`. A reference, not a drop-in.
- **[rehhouari/WeightedChoice](https://github.com/rehhouari/WeightedChoice)** — **MIT** — A ~40-line `WeightedChoice.pick(dict, weight_key)` primitive (correct linear-scan weighted selection). **Don't add as an addon** — copy the one function into `Gacha.gd` and **tighten the types** (the original leaves `weight_key`/`weights` untyped). Caveats: its example weights don't sum to 1.0 (it normalizes internally, so define rates that sum predictably for displayed drop-rates); it uses global `randf()` — refactor to a seeded RNG if ever used inside the sim; add guards for empty/all-zero/negative weights. Value is "not getting the algorithm subtly wrong," with attribution.
- **[EddieDover/praeda](https://github.com/EddieDover/praeda)** — **LGPL-3.0-or-later (weak copyleft) — Rust/GDExtension, NOT GDScript** — **DESIGN REFERENCE ONLY.** The best-structured *spec* for the ARPG affix model: weighted quality tiers, weighted item type/subtype hierarchy, item-level attribute scaling (`base_level`, `level_variance`, `affix_chance`, `scaling_factor`), prefix/suffix affixes. **Reimplement the algorithms** (weighted pick + range roll + level scale — trivial math) as typed GDScript in `systems/data/`; **do NOT copy Rust source** (would impose LGPL on those files) and don't ship the GDExtension (Rust toolchain + per-platform binaries, plus LGPL relinking-compliance burden). APIs/facts aren't copyrightable; source text is.

## Talent tree

The brief wants a PoE-style pan/zoom node web with prerequisites, allocated-glow, and stat grants feeding StatBlock — authored data-driven and decoupled from rendering.

- **[don-tnowe/godot-worldmap-builder](https://github.com/don-tnowe/godot-worldmap-builder)** — **MIT** — **Strongest talent fit** (135★, **already typed** GDScript). `WorldmapView` is the allocatable runtime; author layouts in-editor as `WorldmapGraph` (branching) / `WorldmapPath` (lines/arcs/Bezier). It hands you the hard part: `can_activate`/`can_deactivate`/`set_node_state` (returns cost delta)/`reset()` (respec), with prereqs data-driven via `WorldmapNodeData.dependency_min_state`. **Adapt:** replace its example `SkillStats` resource with our `TalentGrant` resource and fold summed grants from `get_all_nodes()` into our additive/multiplicative StatBlock; route `get_state()`/`load_state(Dictionary)` through SaveManager; reskin `WorldmapStyle` (style_active/can_activate/inactive) with our dark-gothic tokens for the glow. You add the pan/zoom wrapper (Control/ScrollContainer). Low-to-moderate effort.
- **[ToxicStarfall/tree-maps-addon](https://github.com/ToxicStarfall/tree-maps-addon)** — **MIT** — Editor authoring tooling (`TreeMap`/`TreeMapNode` Node2D + directional-connection tools, ~30★, v1.1 Sep 2025). **Author-time only** — lay out the web visually, then read it at runtime; you needn't ship the plugin. **Decouple data from rendering:** define a typed `TalentNode` Resource (id/grid_pos/prerequisites/stat_grants/allocated) and either export the authored scene to `.tres`/JSON or wrap it with a typed adapter. No pan/zoom or in-game allocation UI (build those); straight-line connections only.
- **[Michael-Jalloh/SkillEditor](https://github.com/Michael-Jalloh/SkillEditor)** — **MIT** — Editor plugin using **GraphEdit/GraphNode** that **exports to JSON** — aligns with our data-driven `res://data/` convention. Use as a **content pipeline** to generate talent data files; define your JSON schema (id, grid position, prereq id list, stat grants keyed to our 5 mains, allocated flag) first and adapt its exporter to emit that shape. Tiny/young (8★, 3 commits) — own it; don't depend on it. Not the in-game allocation UI.
- **[Reneator/godot-skill-tree](https://github.com/Reneator/godot-skill-tree)** — **MIT** — **Godot 3.x** (needs porting: `tool`→`@tool`, `export`→`@export`, `config_version` 4→5). A useful **pattern** reference for live in-editor connection-line refresh and a workable prerequisite/unlock model (`previous_nodes_paths`, ANY/ALL `check_unlocked()`, `on_learned` signal, row-pick constraints, active/inactive tints). Drive node defs from `res://data/` feeding a typed `TalentTree.gd`; use the setter-driven line refresh as the renderer only.
- **[SeldonHh/Godot_skill_tree_maker](https://github.com/SeldonHh/Godot_skill_tree_maker)** — **NO LICENSE — all-rights-reserved — DO NOT COPY** — Right engine (Godot 4.4) and validates the Resource-per-node approach, but **conceptual reference only** until/unless the author adds a license. Clean-room reimplement: `class_name TalentNode extends Resource` with typed `@export`s, layout driven by the Resource's `position`, a `TalentTreeController` validating prereqs and applying grants into GameState's StatBlock. No save of allocated state (most of the work for an idle game). Very early (1★).

## Inventory / items

- **[peter-kish/gloot](https://github.com/peter-kish/gloot)** — **MIT** — **Best inventory fit** (927★, Godot 4.4+, typed `class_name`s usable in signatures). Use one `Inventory` for the loot/consumable bag and an `ItemSlot` per equipment-paperdoll slot (helm/chest/gloves/boots/weapon/offhand/2 rings/amulet/belt) + the active pet slot — `ItemSlot` is the right primitive for our slot-based paperdoll (skip `GridConstraint` unless you add a PoE spatial stash). JSON "prototree" prototypes with inheritance map 1:1 onto our data-driven `Item` model; `ItemCountConstraint`/`WeightConstraint` for stash caps; stacking for consumables/shards. **Serialize/deserialize** feeds straight into SaveManager. **Adapt:** map our affix/rarity/level-req onto prototype custom attributes; write a small typed adapter reading those into the StatBlock (GLoot owns data; your code owns stat math); recompute StatBlock + emit via EventBus on equip/unequip. Restyle its `Ctrl*` controls with our design tokens.
- **[gdquest-demos/godot-open-rpg](https://github.com/gdquest-demos/godot-open-rpg)** — **MIT** — Item/inventory conventions and stat-aggregation (how gear/effects feed a computed block) map onto our StatBlock + Item/affix model. Tighten typing on anything imported (the demo isn't guaranteed fully typed). Clone branch `main`.
- **[expressobits/inventory-system](https://github.com/expressobits/inventory-system)** — **MIT — but C++ GDExtension, NOT GDScript (97.4% C++)** — **Architecture/feature reference**, not a line-by-line port. Its logic-vs-UI separation mirrors our sim-vs-render rule; study and reimplement in typed GDScript: equipment-inventory + slot-category restriction (paperdoll), the **crafting/autocraft system → Camp Crafting House**, the loot system, and the item-editor authoring workflow. Avoid the binary dependency (per-platform `.dll`/`.so` build burden + external release cadence break the typed-GDScript constraint). Its grid logic derives from gloot — prefer gloot directly.
- **[guladam/godot_autobattler_course](https://github.com/guladam/godot_autobattler_course)** — **MIT** — Its `data/items/`, `data/units/` `.tres` definitions and **`drag_and_drop.gd`** port nearly directly to the equipment paperdoll + inventory grid; `TooltipHandler` is reusable for item/talent/relic hover tooltips. (Combat caveats under CombatSim.)
- **[alpapaydin/Godot-4-Grid-Inventory-with-Patterns](https://github.com/alpapaydin/Godot-4-Grid-Inventory-with-Patterns)** — **MIT** — **Likely over-scoped:** its headline shaped/rotatable Tetris-stash does **not** match our named-slot paperdoll. Salvageable bits: the drag-and-drop snap/hover flow (strip the matrix math) and its **dictionary serialization** (`load_inventory`/`reconstruct_grid_from_states`) for save/load. Port specific routines typed; don't vendor wholesale. Saves modest time, not a major component.

## UI theme & architecture

We already have the app shell, EventBus, and a design-token system, so these are mostly pattern references and targeted lifts.

- **[TinyTakinTeller/TakinGodotTemplate](https://github.com/TinyTakinTeller/TakinGodotTemplate)** — **MIT** — **Best architecture cross-check** (472★, Godot 4.4, ~99% **typed** GDScript, same author as GodotProjectZero). Prefer studying its Proportional/Aspect-Fit approach over hard-depending on an unmaintained parallel layout paradigm.
- **[BajaTheFrog/godot-event-bus](https://github.com/BajaTheFrog/godot-event-bus)** — **MIT** — **LOW relevance:** we already have an EventBus, and this targets **Godot 3.5.3** with stringly-typed broadcast/subscribe machinery that 4.x native `signal` + `Callable` make obsolete. Vendor a copy (8★, ~4 commits).

At most, re-import a couple of CC0 PNG/SVG nine-patches as throwaway placeholders. The flat-minimal look also conflicts with our gothic mood.

## Full-game references & dev tooling

Install under `res://addons/gut/`, keep it **out of the shipped build** (export-ignore `addons/gut/` + `test/`), run headless via `gut_cmdln` with JUnit XML in CI.
- **[gdquest-demos/godot-open-rpg](https://github.com/gdquest-demos/godot-open-rpg)** — **MIT** — Best overall structural reference for the `scenes/` + `systems/` + `autoload/` split and EventBus conventions. Branch `main`. (System-specific notes under CombatSim/Inventory.)

## License compliance — read before copying anything

Our repo currently has **NO LICENSE file**. That doesn't restrict what *we* may borrow, but it does mean we must be deliberate about third-party obligations and should add our own license/`THIRD_PARTY_NOTICES` before distributing.

- **Permissive (MIT, BSD, Apache, MPL):** reusable in our closed-source Steam build — **but you must retain the original copyright + permission notice** for any file or substantial snippet you copy. Maintain a `THIRD_PARTY_NOTICES.md` / credits file listing every MIT source we lift from (GodotProjectZero, gloot, Lootie, worldmap-builder, savestate, ForgeJSONGD, Gut, the GDQuest demo, the autobattler repos, etc.). No copyleft, no source-disclosure.
- **CC0 / Unlicense (public domain):** copy freely, no attribution required (Themey graphics, Iso-Themes, Clooloo demo). Attribution is still courteous.
- **CC-BY (3.0/4.0):** permissive **with attribution** — covers godot-docs prose and the awesome-godot list. Credit the source if you reproduce non-trivial text/structure.
- **GPL/AGPL copyleft:** none in this set, but the rule stands — **copying GPL code forces The Delve to GPL.** Reference-only is safe; copying is not. (Watch for this when mining awesome-godot.)
- **LGPL-3.0 (weak copyleft) — praeda:** reimplementing its *algorithm/API* in our own GDScript is fine (facts/APIs aren't copyrightable); **copying its Rust source** would put those files under LGPL; **linking the GDExtension** is permitted only with relink-replaceability + source provision — out of scope for v1.
- **NO LICENSE = all rights reserved — NEVER copy:** **LORED-Legacy, EricJPbunny/Godot-Incremental-Game, SeldonHh/Godot_skill_tree_maker.** These are *read-only* design references; do not paste, port, or vendor a single line. If code from one ever becomes essential, contact the author for explicit permission first.
- **Mixed-license repos — partition carefully:** **GodotProjectZero** is MIT *for code* but **CC-BY-NC-SA for art/music/shaders/writing** — the NC (non-commercial) term is **incompatible with a paid Steam release**, so lift code patterns only and rebuild all assets ourselves. **Themey** is MIT code + CC0 graphics. The GitHub API "NOASSERTION/other" SPDX flag on GodotProjectZero reflects this mix, **not** unlicensed code.
- **Honest uncertainties:** a few small repos didn't surface exact typing/version details on inspection — verify static typing per-file before lifting (GDQuest demo, Maaack/Takin templates), and smoke-test version compatibility on our exact Godot stable (Lootie 4.3/4.4; Godot-GameGUI tested only on 4.2-dev; theme `.tres` slot coverage across 4.4→4.6).
